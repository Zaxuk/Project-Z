# Spring
## 事务的ACID四大原则
- Atomicity 原子性
  - 事务要么全部成功，要么全部失败
- Consistency 一致性
  - 事务执行前后，数据库的一致性得到维护
  - 例如：订单金额不能小于0
- Isolation 隔离性
  - 事务执行时，其他事务不能访问到未提交的数据
  - 例如：用户A在支付订单时，用户B不能查看到用户A的订单金额
  - 例如：用户A在支付订单时，用户B不能查看到用户A的订单状态
- Durability 持久性
  - 事务执行后，数据库的持久性得到维护
  - 例如：用户A支付订单后，订单金额从用户A的账户中扣除
  - 例如：用户A支付订单后，订单状态从待支付变为已支付

## Spring事务的传播行为
- PROPAGATION_REQUIRED
  - 默认值
  - 如果当前事务存在，加入当前事务
  - 如果当前事务不存在，创建新事务
- PROPAGATION_REQUIRES_NEW
  - 如果当前事务存在，挂起当前事务
  - 如果当前事务不存在，创建新事务
- PROPAGATION_NESTED
  - 如果当前事务存在，创建嵌套事务
  - 如果当前事务不存在，创建新事务
- PROPAGATION_MANDATORY
  - 如果当前事务存在，加入当前事务
  - 如果当前事务不存在，抛出异常
- PROPAGATION_SUPPORTS
  - 如果当前事务存在，加入当前事务
  - 如果当前事务不存在，不创建事务
- PROPAGATION_NOT_SUPPORTED
  - 如果当前事务存在，挂起当前事务
  - 如果当前事务不存在，不创建事务
- PROPAGATION_NEVER
  - 如果当前事务存在，抛出异常
  - 如果当前事务不存在，不创建事务

## Spring事务的隔离级别
- ISOLATION_DEFAULT
  - 默认值
  - 数据库默认的隔离级别
  - 例如：MySQL默认的隔离级别是READ_COMMITTED
  - 例如：Oracle默认的隔离级别是READ_UNCOMMITTED
- ISOLATION_READ_UNCOMMITTED
  - 最低隔离级别
  - 读取未提交的数据
  - 可能存在脏读、不可重复读、幻读问题
  - 例如：用户A在支付订单时，用户B可以查看到用户A的订单金额
  - 例如：用户A在支付订单时，用户B可以查看到用户A的订单状态
- ISOLATION_READ_COMMITTED
  - 读取已提交的数据
  - 可能存在不可重复读、幻读问题
  - 例如：用户A在支付订单时，用户B不能查看到用户A的订单金额
  - 例如：用户A在支付订单时，用户B不能查看到用户A的订单状态
- ISOLATION_REPEATABLE_READ
  - 重复读取已提交的数据
  - 可能存在幻读问题
  - 例如：用户A在支付订单时，用户B不能查看到用户A的订单金额
  - 例如：用户A在支付订单时，用户B不能查看到用户A的订单状态
- ISOLATION_SERIALIZABLE
  - 最高隔离级别
  - 串行执行

# 架构
## 分布式事务
- 2PC
  - 两阶段提交协议
  - 基于事务协调器 TM，分为「准备阶段 Prepare + 提交阶段 Commit」
  - 准备阶段：TM 通知所有参与者 RM 执行事务、锁定资源，返回是否可提交
  - 提交阶段：所有参与者成功则通知提交，任意一个失败则通知回滚
- 3PC
  - 三阶段提交协议
  - 基于 2PC 基础上增加 CanCommit 阶段，分为「CanCommit+PreCommit+DoCommit」
  - CanCommit 仅询问可行性，不锁定资源；同时引入超时机制，降低同步阻塞范围
- TCC
  - Try-Confirm-Cancel
  - 业务层的两阶段补偿型事务，分为三个阶段：
  - 1. Try：资源检查、预占 / 锁定；
  - 2. Confirm：确认提交，必须幂等；
  - 3. Cancel：回滚释放资源，必须幂等
